🏛️Chomsky Hierarchy ⭐ Always in Exam

The Chomsky Hierarchy classifies formal languages into 4 types based on the automaton that recognizes them and the production rules of their grammar.

Type 0 — Recursively Enumerable Language (REL) · Recognized by: Turing Machine · Production: αAβ → γ
Type 1 — Context-Sensitive Language (CSL) · Recognized by: Linear Bounded Automata (LBA) · Production: αAβ → αγβ
Context-Free Language (CFL) · Recognized by: PDA (Pushdown Automaton) · Production: A → γ
Type 3 — Regular Language · Recognized by: DFA / NFA / FA · Production: A → aB or A → a
⭐ Exam Tip: Remember the strict subset relation: Regular ⊂ CFL ⊂ CSL ⊂ REL. Also note: every regular language is CFL, but NOT vice versa. L = {aⁿbⁿ | n ≥ 1} is CFL but NOT regular.
📋Full Chomsky Table
TypeLanguage ClassGrammar Production RuleRecognizing MachineClosure Under
Type 3 Regular A → aB or A → a DFA / NFA / ε-NFA Union, Concat, Kleene*, Complement, Intersection, Reversal
Type 2 Context-Free A → γ (A is single non-terminal) Non-deterministic PDA Union, Concat, Kleene*; NOT complement, NOT intersection
Type 1 Context-Sensitive αAβ → αγβ, |γ| ≥ 1 Linear Bounded Automaton (LBA) Union, Concat, Kleene*, Complement, Intersection
Type 0 Recursively Enumerable Any production: α → β Turing Machine Union, Concat, Kleene*, Intersection; NOT complement
⭐ Frequently Asked: Closure Properties — CFL is NOT closed under complement and intersection. CFL ∩ Regular = CFL (important!). Two CFLs intersection is NOT necessarily CFL.
📌Key Language Examples — Identify the Type

🔵 Regular (Type 3)

  • L = {aⁿ | n ≥ 0} = a*
  • L = {w | w has even number of a's}
  • L = {w ∈ {a,b}* | w starts with a}
  • L = {(ab)ⁿ | n ≥ 1}
  • L = strings over {0,1} with at least two 1s

🟢 CFL but NOT Regular (Type 2)

  • L = {aⁿbⁿ | n ≥ 1}
  • L = {aⁿbⁿcⁿ... wait — this is NOT CFL!}
  • L = {wwᴿ | w ∈ {a,b}*} (palindromes)
  • L = {w | w is valid arithmetic expression}
  • L = {aⁿb²ⁿ | n ≥ 1}

🟡 CSL but NOT CFL (Type 1)

  • L = {aⁿbⁿcⁿ | n ≥ 1} ← Very frequently tested!
  • L = {ww | w ∈ {a,b}*}
  • L = {aⁿbⁿcⁿdⁿ | n ≥ 1}
  • L = {1ⁿ | n is a perfect square}

🔴 Recursively Enumerable / Not RE

  • RE: All recursively decidable languages
  • RE: Halting problem language (accepts but may not halt)
  • NOT RE: Complement of Halting Problem
  • NOT RE: Languages TM cannot recognize at all
⭐ Trick: L = {aⁿbⁿcⁿ | n ≥ 1} is CSL, NOT CFL. To prove non-CFL, use the Pumping Lemma for CFL.
⚙️DFA — Deterministic Finite Automaton

A DFA is a 5-tuple: M = (Q, Σ, δ, q₀, F)

Components

  • Q — finite set of states
  • Σ — finite input alphabet
  • δ: Q × Σ → Q — transition function
  • q₀ ∈ Q — start state
  • F ⊆ Q — set of accepting states

Key Properties

  • Exactly ONE transition per state per symbol
  • Total function — defined for all (state, symbol)
  • Processes each input symbol exactly once
  • Deterministic — no choice
  • Minimum DFA is unique for each regular language
⭐ Exam Favorite: Number of states in minimum DFA for a given language / RE is frequently asked. Also: number of transitions = |Q| × |Σ|.
🔄NFA to DFA Conversion Very Important

NFA = (Q, Σ, δ, q₀, F) where δ: Q × Σ → 2^Q (powerset)

Key Fact: If NFA has n states, the equivalent DFA can have at most 2ⁿ states. But usually far fewer are reachable.

ε-NFA to NFA Conversion

  • Compute ε-closure(q) = all states reachable from q via ε-transitions only
  • New start state = ε-closure(q₀)
  • δ'(S, a) = ε-closure(⋃{δ(q,a) | q ∈ S})
  • A state S is accepting if S ∩ F ≠ ∅
⭐ Exam Tip: ε-NFA, NFA, DFA — all have EQUAL POWER. They all recognize exactly the Regular Languages. Adding ε-transitions does NOT increase power.
🔡Regular Expressions Very Important

Basic Operations

r + s → Union (r OR s) r · s → Concatenation r* → Kleene star (zero or more r) r+ → r · r* (one or more) r? → r + ε (zero or one)

Precedence (High → Low)

1. Kleene Star (*) 2. Concatenation (·) 3. Union (+) So: a+bc* = a + (b(c*))

Important Identities

∅* = ε ε* = ε (r*)* = r* r + ∅ = r r · ε = r r · ∅ = ∅ r + r = r (r+s)* = (r*s*)* = (r*+s*)* (r·s)* ≠ r*·s* but (rs)* r = r(sr)*
⭐ Arden's Theorem: If X = Q + RX then X = QR* (provided ε ∉ R). Used to convert DFA → Regular Expression using linear equations.
🔵Pumping Lemma for Regular Languages Proof Tool

Theorem: If L is regular, then ∃ pumping length p ≥ 1 such that every string w ∈ L with |w| ≥ p can be written as w = xyz where:

1. |xy| ≤ p (y is within first p characters) 2. |y| ≥ 1 (y is non-empty) 3. ∀ i ≥ 0: xyⁱz ∈ L (pumping y keeps string in L)

How to use it (to prove L is NOT regular):

  1. Assume L is regular with pumping length p
  2. Choose a string w ∈ L with |w| ≥ p (cleverly!)
  3. Show that for ALL ways to split w = xyz (with conditions), some xyⁱz ∉ L
  4. This contradicts the lemma → L is not regular
⭐ Classic Example: L = {aⁿbⁿ | n ≥ 1}. Choose w = aᵖbᵖ. Since |xy| ≤ p, y = aᵏ (k ≥ 1). Pump i=0: xz = aᵖ⁻ᵏbᵖ — unequal counts → NOT in L. So L is not regular. ✓
⚠️ Important: Pumping Lemma can only DISPROVE regularity. Satisfying it does NOT prove a language IS regular!
⚡DFA Minimization (Myhill-Nerode)

The minimum DFA has the fewest states. Algorithm:

  1. Remove unreachable states
  2. Mark all {accepting, non-accepting} pairs as distinguishable
  3. Iteratively mark: (p, q) distinguishable if ∃ a ∈ Σ such that (δ(p,a), δ(q,a)) is already marked
  4. Merge all unmarked (equivalent) state pairs
⭐ Myhill-Nerode Theorem: Number of states in min DFA = number of equivalence classes of the right-invariant equivalence relation ~ defined by: x ~ y iff ∀z: xz ∈ L ↔ yz ∈ L. If this has infinitely many classes → L is NOT regular.
🌳Context-Free Grammar (CFG)

A CFG is a 4-tuple: G = (V, T, P, S)

Components

  • V — Variables (non-terminals)
  • T — Terminals
  • P — Production rules: A → α
  • S ∈ V — Start symbol

Types of Derivation

  • Leftmost: always replace leftmost non-terminal
  • Rightmost: always replace rightmost non-terminal
  • Parse Tree: unique if grammar is unambiguous
⭐ Ambiguous Grammar: A grammar is ambiguous if some string has MORE THAN ONE parse tree (or more than one leftmost derivation). Inherently ambiguous CFL: no unambiguous grammar exists for it.
📝Normal Forms of CFG Important

Chomsky Normal Form (CNF)

Every production is either:

A → BC (two non-terminals) A → a (single terminal) S → ε (only if ε ∈ L)

Steps to convert: Eliminate ε-rules → Eliminate unit rules → Eliminate useless symbols → Convert remaining rules

Use: CYK parsing algorithm requires CNF. Every CFL has a CNF grammar.

Greibach Normal Form (GNF)

Every production is of the form:

A → aα where a ∈ T, α ∈ V*

Every right-hand side starts with a terminal followed by zero or more non-terminals.

Use: GNF is useful for constructing PDAs. Each derivation step consumes exactly one terminal.
📚Pushdown Automaton (PDA)

PDA = (Q, Σ, Γ, δ, q₀, Z₀, F)

Components

  • Q — states
  • Σ — input alphabet
  • Γ — stack alphabet
  • δ — transition
  • q₀ — start state
  • Z₀ — start stack symbol
  • F — final states

Acceptance Modes

  • Final State: accepted when in final state after reading input
  • Empty Stack: accepted when stack is empty after reading input
  • Both modes are equivalent in power

DPDA vs NPDA

  • NPDA is more powerful than DPDA
  • DPDA accepts Deterministic CFLs
  • {aⁿbⁿ} — accepted by DPDA
  • {wwᴿ} — requires NPDA
  • All regular languages are DCFL
⭐ Classic PDA for aⁿbⁿ: Push a's when reading a, pop one a for each b read. If stack empty when all input read → accept.
🔵Pumping Lemma for CFLs Important

If L is CFL with pumping length p, then every w ∈ L with |w| ≥ p can be written as w = uvxyz where:

1. |vxy| ≤ p (middle part bounded) 2. |vy| ≥ 1 (at least one of v,y is non-empty) 3. ∀ i ≥ 0: uvⁱxyⁱz ∈ L (pump both v and y together)
⭐ Classic Proof: L = {aⁿbⁿcⁿ | n ≥ 1} is NOT CFL. Choose w = aᵖbᵖcᵖ. Any split uvxyz: pumping v and y can at most affect 2 types of characters (a,b or b,c or a alone, etc.). Can't keep all three counts equal → contradiction.

🖥️Turing Machine

TM = (Q, Σ, Γ, δ, q₀, B, F) where B is the blank symbol

Components

  • Q — states
  • Σ — input alphabet (Σ ⊂ Γ)
  • Γ — tape alphabet
  • δ: Q × Γ → Q × Γ × {L, R}
  • q₀ — start state
  • B — blank symbol (B ∈ Γ, B ∉ Σ)
  • F — set of final/halt states

Important Variants

  • Multi-tape TM: Same power as single-tape
  • Non-deterministic TM: Same power as deterministic
  • Universal TM: Can simulate any TM
  • Linear Bounded Automaton (LBA): TM restricted to input length tape
Church-Turing Thesis: Any algorithm computable by any mechanical means can be computed by a Turing Machine. This is a thesis (hypothesis), not a provable theorem.
⚖️Decidability Very Frequently Asked

Decidable (Recursive) Languages

  • Membership for Regular Languages
  • Membership for CFL (using CYK)
  • Emptiness of Regular Language
  • Equivalence of two DFAs
  • Equivalence of two Regular Expressions
  • Finiteness of Regular Language
  • Every CFL decidable by TM

Undecidable Problems

  • Halting Problem — Does TM M halt on input w?
  • Emptiness of RE Language
  • Equivalence of two CFGs
  • Ambiguity of CFG
  • Post Correspondence Problem (PCP)
  • Whether L(TM) = Σ*
  • Whether two TMs accept same language
⭐ Key: A language is Recursively Enumerable (RE) if some TM accepts it (may loop on non-input). It is Recursive (Decidable) if some TM always halts and decides membership. Recursive ⊂ RE.
❌Halting Problem — Undecidability Proof

The Halting Problem: Given TM M and input w, does M halt on w?

Proof by Contradiction (Diagonalization): 1. Assume H(M, w) is a decider for halting problem 2. Construct D(M): if H(M,⟨M⟩) = "halt" → loop forever if H(M,⟨M⟩) = "loop" → halt 3. Run D on itself: D(⟨D⟩) - If D halts → D should loop → Contradiction! - If D loops → D should halt → Contradiction! Therefore, no such H exists. Halting Problem is UNDECIDABLE.
🎯Rice's Theorem ⭐ Most Important
Rice's Theorem: Any non-trivial property of the language of a Turing Machine is undecidable.

Property P is trivial if: either ALL TMs satisfy P, or NO TM satisfies P.

Property P is non-trivial if: some TMs satisfy it and some don't.

Examples of UNDECIDABLE by Rice's Theorem: • Is L(M) empty? (non-trivial property) • Is L(M) finite? (non-trivial property) • Is L(M) a regular language? (non-trivial property) • Does M accept string "010"? (non-trivial property) • Is L(M) = L(M₂) for given M₂?
⭐ Exam Trick: Whenever a question asks "Is there a TM/algorithm that decides whether a TM's language has property X?" — if X is non-trivial (not all or nothing), the answer is NO by Rice's Theorem!
🔗Reducibility

A ≤ₘ B (A reduces to B)

  • If A reduces to B and B is decidable → A is decidable
  • If A reduces to B and A is undecidable → B is undecidable
  • Halting Problem ≤ₘ many other problems

Important Results

  • Regular ⊂ DCFL ⊂ CFL ⊂ Recursive ⊂ RE
  • Complement of RE language may not be RE
  • Complement of Recursive = Recursive
  • If L and L̄ are both RE → L is Recursive
🔧Phases of a Compiler ⭐ Always Asked
1. Lexical Analysis (Scanner)
Input: Source code → Output: Token stream
Tool: Regular Expressions → DFA
Detects: Lexical errors
2. Syntax Analysis (Parser)
Input: Tokens → Output: Parse Tree / AST
Tool: CFG → Parsing Algorithms
Detects: Syntax errors
3. Semantic Analysis
Input: Parse Tree → Output: Annotated AST
Checks: Type checking, scope rules
Detects: Semantic errors
4. Intermediate Code Generation
Input: Annotated AST → Output: IR (3-address code)
Machine-independent representation
5. Code Optimization
Input: IR → Output: Optimized IR
Improves speed/memory (optional phase)
6. Code Generation
Input: Optimized IR → Output: Target Machine Code
Register allocation, instruction selection

Symbol Table

Used by ALL phases. Stores identifiers, types, scopes, memory locations.

Error Handler

Active across all phases. Each phase detects and reports different kinds of errors.

Two Passes

Front End: Phases 1–3 (machine-independent, analysis)
Back End: Phases 4–6 (machine-dependent, synthesis)

Contents

  • Data type
  • Scope / visibility
  • Memory location / offset
  • Number of parameters (for functions)

🔢FIRST Sets — Rules & Algorithm ⭐ Most Important

FIRST(α) = set of terminals that begin strings derivable from α. Also ε if α ⟹* ε.

Rules for FIRST(X): ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 1. If X is a terminal: FIRST(X) = {X} 2. If X → ε exists: Add ε to FIRST(X) 3. If X → Y₁Y₂...Yₖ: • Add FIRST(Y₁) − {ε} to FIRST(X) • If ε ∈ FIRST(Y₁): Add FIRST(Y₂) − {ε} • If ε ∈ FIRST(Y₁) and FIRST(Y₂): Add FIRST(Y₃)... • If ε ∈ FIRST(Yᵢ) for ALL i: Add ε to FIRST(X)
⭐ In Exam: Given a grammar, compute FIRST for each non-terminal. This is tested almost in every UGC NET exam. Practice 5-6 examples.
🔢FOLLOW Sets — Rules & Algorithm ⭐ Most Important

FOLLOW(A) = set of terminals that can appear immediately to the RIGHT of A in any sentential form. Never contains ε.

Rules for FOLLOW(A): ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 1. Place $ in FOLLOW(S) [S = start symbol] 2. If B → αAβ: Add FIRST(β) − {ε} to FOLLOW(A) 3. If B → αA or B → αAβ where ε ∈ FIRST(β): Add FOLLOW(B) to FOLLOW(A) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Note: Apply rules repeatedly until no FOLLOW set changes.
⚠️ Common Mistake: FOLLOW never contains ε! If ε ∈ FIRST(β), you add FOLLOW(B), not ε itself.
📘Worked Example: FIRST & FOLLOW
Grammar: E → T E' E' → + T E' | ε T → F T' T' → * F T' | ε F → ( E ) | id FIRST: FIRST(F) = { (, id } FIRST(T') = { *, ε } FIRST(T) = FIRST(F) = { (, id } FIRST(E') = { +, ε } FIRST(E) = FIRST(T) = { (, id } FOLLOW: FOLLOW(E) = { ), $ } FOLLOW(E') = FOLLOW(E) = { ), $ } FOLLOW(T) = FIRST(E') − {ε} ∪ FOLLOW(E') = { +, ), $ } FOLLOW(T') = FOLLOW(T) = { +, ), $ } FOLLOW(F) = FIRST(T') − {ε} ∪ FOLLOW(T') = { *, +, ), $ }
⭐ This exact grammar appears in UGC NET very often! Memorize FIRST and FOLLOW sets for this expression grammar.
📊LL(1) Parsing Table Construction

For each production A → α:

1. For each terminal a ∈ FIRST(α) − {ε}: Add (A → α) to M[A, a] 2. If ε ∈ FIRST(α): For each b ∈ FOLLOW(A): Add (A → α) to M[A, b] (If $ ∈ FOLLOW(A), add to M[A, $] too)

LL(1) Grammar Conditions

  • No Left Recursion (direct or indirect)
  • No ambiguity
  • No two entries in same cell of parsing table
  • Left Factored when needed

Left Recursion Removal

A → Aα | β
Becomes:
A → β A'
A' → α A' | ε
⭐ Key: A grammar is LL(1) if and only if its parsing table has NO conflicts (no cell has more than one entry). If M[A,a] has two productions → grammar is NOT LL(1).

⚡LR Parsing — Overview ⭐ Very Important

LR parsers are bottom-up parsers that use a stack + parsing table. "L" = left to right scan, "R" = rightmost derivation in reverse.

LR Parsing Table

  • ACTION table: terminal columns → shift/reduce/accept/error
  • GOTO table: non-terminal columns → state number
  • States come from LR automaton

Power Ordering

LR(1) = CLR(1)
  ⊃ LALR(1)
    ⊃ SLR(1)
      ⊃ LR(0)

More states = more power to resolve conflicts.

All LL(1) grammars are LR(1) but NOT vice versa. LR parsing is more powerful than LL parsing.
📐LR(0) Items & Automaton
LR(0) Item: A → α • β The dot (•) marks how much of the production has been seen. Closure operation: • For each item A → α • Bβ in the set, add all items B → • γ for each production B → γ GOTO(I, X): • Move the dot past X in all items that have • before X • Take closure of result
⭐ Augmented Grammar: Add production S' → S to the grammar before building LR automaton. The start item is S' → •S.

🚨Parsing Conflicts

Shift-Reduce Conflict

When parser can either shift next input OR reduce current stack — both valid actions for same state and input symbol.

Dangling else problem is a classic S/R conflict example.

Reduce-Reduce Conflict

When parser can reduce using two different productions for same state and input symbol. Indicates grammar ambiguity or inadequate lookahead.

LALR(1) can have R/R conflicts that CLR(1) doesn't have.

⭐ Exam Tip: If a grammar is ambiguous → it can NEVER be parsed without conflict by ANY LR parser. Unambiguous grammar may still have conflicts in LR(0) or SLR(1) but might be fine with CLR(1).
🗂️Intermediate Code Representations Important

Three-Address Code (TAC)

  • x = y op z
  • x = op y
  • x = y
  • goto L
  • if x goto L
  • param x, call p, n

Quadruples

(op, arg1, arg2, result)

e.g. a = b + c * d:
(*, c, d, t1)
(+, b, t1, t2)
(=, t2, _, a)

Triples

(op, arg1, arg2)
Result referenced by index

e.g. a = b + c * d:
(0) (*, c, d)
(1) (+, b, (0))
(2) (=, a, (1))

Indirect triples: use pointer array to allow reordering

⭐ Key Difference: Quadruples have explicit result field (easier to optimize). Triples reference by position (compact but harder to reorder). Indirect Triples solve the reordering problem.
🎛️Syntax Directed Translation (SDT)

Attribute Grammar

  • Synthesized attributes: computed from children's values (bottom-up, S-attributed)
  • Inherited attributes: passed down from parent/siblings (top-down)
  • S-attributed: only synthesized → can be evaluated during LR parsing
  • L-attributed: synthesized + inherited from LEFT siblings → can be evaluated during LL parsing

Evaluation Order

S-attributed ⊂ L-attributed Both evaluated in one pass if dependencies form a DAG (not cycle). S-attributed → evaluated bottom-up L-attributed → evaluated left-to-right depth-first traversal
⭐ Key: Every S-attributed grammar is also L-attributed. L-attributed is used with top-down parsers.
⚡Code Optimization Techniques Important
Machine-Independent Optimizations

Local Optimizations

  • Common Subexpression Elimination (CSE): Avoid recomputing same expression
  • Constant Folding: Evaluate constant expressions at compile time (3+4 → 7)
  • Constant Propagation: Replace variable with its constant value
  • Dead Code Elimination: Remove unreachable or unused code
  • Copy Propagation: Replace x = y; use x → use y directly

Loop Optimizations

  • Loop Invariant Code Motion: Move code that doesn't change in loop to outside
  • Induction Variable Elimination: Simplify variables that change by constant amount each iteration
  • Loop Unrolling: Reduce loop overhead by duplicating body
  • Strength Reduction: Replace costly op with cheaper (x*2 → x+x)
  • Loop Fusion: Combine two adjacent loops into one
Machine-Dependent Optimizations

Peephole Optimization

  • Examines small window ("peephole") of instructions
  • Remove redundant loads/stores
  • Simplify algebraic expressions (x + 0 → x)
  • Replace jump chains
  • Use machine idioms (INC instead of ADD 1)

Register Allocation

  • Assign variables to CPU registers (limited resource)
  • Graph coloring approach: variables conflict if simultaneously live
  • Spilling: store to memory when registers full
⭐ DAG (Directed Acyclic Graph) is used to represent Basic Blocks for optimization. It automatically identifies common subexpressions and eliminates redundant computations.
🗃️Runtime Storage Organization

Activation Record / Stack Frame

  • Return value
  • Actual parameters
  • Control link (dynamic link — points to caller's AR)
  • Access link (static link — for lexical scoping)
  • Saved machine state
  • Local variables
  • Temporaries

Storage Allocation Types

  • Static: allocated at compile time, fixed addresses. Global variables, static local vars.
  • Stack: allocated/freed on function call/return. Local variables, parameters.
  • Heap: dynamic allocation at runtime (malloc/new). Managed by programmer or GC.
